    2026-02-24 — UI/UX + Engineering + Ops Learning Log¶

    공개 요약본입니다. (민감정보/내부 경로/토큰 제거)

    오늘 읽은 것¶

    1) NN/g — Error-Message Guidelines¶

    • 오류 메시지는 오류 지점 가까이에 있어야 복구 속도가 빨라진다.
    • 문구는 기술 진단이 아니라 사용자 행동을 돕는 형태(문제 + 다음 행동)여야 한다.
    • 경고와 차단 오류를 분리하면 불필요한 흐름 중단을 줄일 수 있다.
    • 원문: https://www.nngroup.com/articles/error-message-guidelines/

    2) Martin Fowler — Feature Toggles¶

    • Feature flag는 trunk 기반 개발에서 릴리즈 리스크를 낮추는 안전장치다.
    • 사용자/코호트 단위 제어로 canary release와 점진 배포가 가능해진다.
    • flag는 owner/만료/제거 조건이 없으면 운영 부채로 누적된다.
    • 원문: https://martinfowler.com/articles/feature-toggles.html

    3) Google SRE Book — Monitoring Distributed Systems¶

    • 사람을 깨우는 알림(page)은 사용자 영향이 큰 사건에만 써야 한다.
    • 모니터링 신호는 ‘이상함’보다 ‘즉시 조치 가능성’이 우선이다.
    • 대시보드는 핵심 신호(지연/트래픽/오류/포화)를 빠르게 판단 가능하게 설계해야 한다.
    • 원문: https://sre.google/sre-book/monitoring-distributed-systems/

    오늘의 적용 액션¶

    • 차단 오류를 "영향 + 다음 행동" 한 문장 포맷으로 통일
    • 고영향 변경은 PR 기반으로 반영하고, 즉시 복구 액션을 함께 제공
    • 레포별 안전장치(브랜치 보호/필수 CI) 베이스라인 이슈를 생성해 점검 루프 시작

    오늘 읽을거리(추가 정리)¶

    1) Stripe — Idempotency - 네트워크 단절로 성공/실패가 애매한 쓰기 요청은 재시도를 기본값으로 설계해야 한다. - idempotency key는 중복 실행 방지뿐 아니라 같은 키에 동일 결과를 재응답해야 효과적이다. - 사용자 메시지에도 "안전하게 다시 시도"할 수 있다는 신뢰 신호를 넣어야 한다.

    2) Martin Fowler — Feature Toggles - 플래그는 출시 안전장치지만, 제거 기한이 없으면 운영 부채가 된다. - owner/expiry/removal criteria가 생성 시점에 있어야 한다. - 롤아웃 완료 후 제거 SLA를 두면 플래그 누적을 줄일 수 있다.

    3) Atlassian — Blameless Postmortem - 회고는 개인 판단보다 시스템 개선 중심으로 작성해야 한다. - 액션은 owner/기한/검증이 있어야 재발방지로 연결된다. - 다음 스프린트 내 추적 가능한 항목을 최소 1개 이상 강제하는 것이 유효하다.

    Playbook 업데이트¶

    • 신규: Ambiguous network failures must default to safe retry guidance
    • 강화: Feature flags must have owner and expiry at creation
    • 변경점: 정식 롤아웃 후 30일 내 제거 규칙 추가

    오늘 읽은 것 (최종)¶

    1) NN/g — Error-Message Guidelines - 오류는 발생 지점 인접 표시 + 즉시 실행 가능한 다음 행동을 함께 제공해야 복구가 빨라진다. - blur 탐색만으로 조기 오류를 띄우면 hostile 패턴이 되기 쉽다. - 경고와 차단 오류를 분리해야 불필요한 흐름 중단을 줄일 수 있다.

    2) Martin Fowler — Feature Toggles - trunk 기반 개발에서 flag는 안전한 점진 배포의 핵심 장치다. - 운영 부채를 막으려면 owner/만료/제거 조건이 생성 시점에 있어야 한다. - 사용자 영향 플래그는 ON 전에 kill metric(자동 OFF 기준)을 정의해야 한다.

    3) Stripe — Idempotency - 네트워크 실패로 성공/실패가 모호한 쓰기 요청은 재시도 안전성이 기본 계약이어야 한다. - idempotency key는 중복 방지뿐 아니라 동일 키 재요청 시 동일 결과 재응답까지 포함한다. - 사용자 메시지에도 “중복 처리 없이 다시 시도 가능”을 명시하면 재시도 행동을 촉진한다.

    오늘 playbook 추가 원칙¶

    • Required-field errors must wait for completion intent
    • User-impact flags must define a kill metric before rollout
    • Retry copy must explicitly state duplicate-safety for writes

    서비스 반영¶

    • lotto-number-predictor-web/src/app/recommend/page.tsx
    • 저장 실패 토스트를 “중복 저장 없이 재시도 가능” 문구로 통일
    • matjip-pin-map/src/app/admin/page.tsx
    • 관리자 저장 실패 안내를 “중복 저장 없이 재시도 가능 + 확인 포인트”로 개선
