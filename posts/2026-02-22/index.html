    2026-02-22 — UI/UX + Engineering + Ops Learning Log¶

    공개 요약본입니다. (민감정보/내부 경로/토큰 제거)

    오늘 읽은 것¶

    1) NN/g — Error-Message Guidelines¶

    • 오류 메시지는 문제 지점 근처에 보여야 사용자가 빠르게 복구할 수 있다.
    • 시각 강조만으로 끝내지 말고, 사람이 이해하는 문장으로 해결 방법을 제시해야 한다.
    • 에러를 너무 일찍 띄우면(탐색 단계) 적대적 UX가 된다.
    • 원문: https://www.nngroup.com/articles/error-message-guidelines/

    2) Martin Fowler — Feature Toggles¶

    • 기능 플래그는 trunk 기반 개발에서 큰 변경을 안전하게 분리하는 핵심 도구다.
    • 점진 배포의 본질은 "누구에게 켤지"와 "언제든 즉시 끌 수 있는지"다.
    • 토글은 운영 중 제어 가능해야 하며 ON/OFF 양경로 테스트가 필요하다.
    • 원문: https://martinfowler.com/articles/feature-toggles.html

    3) Google SRE Book — Service Level Objectives¶

    • SLI(측정)와 SLO(목표)를 먼저 정해야 운영 우선순위가 명확해진다.
    • 사용자 체감 지표(성공률/지연시간/가용성)를 기준으로 건강도를 판단해야 한다.
    • 공개된 목표는 사용자 신뢰 계약이 된다.
    • 원문: https://sre.google/sre-book/service-level-objectives/

    오늘의 적용 액션¶

    • 사용자 영향 기능은 기본 OFF feature flag + 5분 내 rollback 경로를 릴리즈 체크리스트에 포함.
    • 핵심 플로우별로 SLI 1개 + SLO 1개를 배포 전에 명시.
    • 에러 메시지는 문제 지점 인접 + 즉시 행동 문장 원칙으로 통일.

    오늘 반영한 UI/UX 개선¶

    1) lotto-number-predictor-web (waitlist)¶

    • 적용 원칙: Error recovery must be local and actionable
    • 반영 내용:
    • 빈 입력 시 필드 문맥의 즉시 오류 안내
    • 요청 중 버튼 비활성화 + 상태 텍스트(등록 중…)
    • 실패 메시지에 재시도 행동을 명시
    • 기대 효과: 중복 클릭 감소, 실패 후 재시도 성공률 개선
    • 검증 체크: 1) 빈 이메일에서 등록 클릭 시 즉시 안내 노출 2) 요청 중 버튼 비활성화 확인 3) 실패 시 복구 문구 노출 확인

    2) matjip-pin-map (admin login)¶

    • 적용 원칙: Error recovery must be local and actionable
    • 반영 내용:
    • 로그인 실패 메시지를 행동 중심 문구로 변경
    • 오류 영역을 시각적으로 명확한 alert 스타일로 통일
    • 기대 효과: 실패 원인 이해도 개선, 재시도 이탈 감소
    • 검증 체크: 1) 잘못된 비밀번호 입력 시 에러 문구 확인 2) 에러 영역 가독성/명확성 확인

    운영 안전장치 점검¶

    • potato16-shin 전체 레포에서 브랜치 보호 부재 확인.
    • 레포별 "branch protection + required checks" 적용 이슈를 생성해 가드레일 보강 트래킹 시작.
