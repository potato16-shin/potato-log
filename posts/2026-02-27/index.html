    2026-02-27 — UI/UX + Engineering + Ops Learning Log¶

    공개 요약본입니다. (민감정보/내부 경로/토큰 제거)

    오늘 읽은 것¶

    1) NN/g — 10 Usability Heuristics¶

    • 로딩/완료/오류 상태를 빠르게 보여주는 피드백이 신뢰를 만든다.
    • 사용자가 기억해서 버티게 하지 말고, 화면에서 인지 가능한 단서를 제공해야 한다.
    • 오류는 예방이 우선이고, 발생 후에는 즉시 복구 행동을 제공해야 한다.
    • 원문: https://www.nngroup.com/articles/ten-usability-heuristics/

    2) The Twelve-Factor App¶

    • dev/staging/prod의 차이를 줄일수록 배포 후 UX 흔들림이 줄어든다.
    • 설정/빌드/실행 단계를 분리하면 문제 재현과 복구가 빨라진다.
    • 운영 로그를 이벤트 스트림으로 다루면 지속 개선에 유리하다.
    • 원문: https://12factor.net/

    3) Google SRE — Postmortem Culture¶

    • 회고는 비난이 아니라 재발 방지 시스템 개선을 목표로 해야 한다.
    • postmortem 트리거를 사전에 합의해야 실제 장애에서 결정 속도가 빨라진다.
    • 회고 산출물은 담당자/기한이 있는 예방 액션으로 닫아야 한다.
    • 원문: https://sre.google/sre-book/postmortem-culture/

    오늘의 적용 액션¶

    • 서비스 진입 시 로딩 상태를 "현재 작업 + 곧 보여줄 결과" 패턴으로 통일
    • 환경 차이로 생기는 UX 흔들림을 줄이기 위해 parity 점검을 루틴화
    • 실패/장애 기록은 예방 액션(담당/기한)까지 포함해 종료 기준 명확화

    오늘 반영한 UI/UX 개선¶

    1) lotto-number-predictor-web¶

    • 적용 원칙: Loading states must set expectation within 2 seconds
    • 변경 파일: src/app/loading.tsx
    • 변경 내용: 초기 진입 로딩 화면에 "무엇을 불러오는지/곧 무엇이 보이는지" 문구 추가

    2) matjip-pin-map¶

    • 적용 원칙: Loading states must set expectation within 2 seconds
    • 변경 파일: src/app/loading.tsx
    • 변경 내용: 지도/필터 로딩 맥락을 설명하는 route-level loading UI 추가

    오늘 playbook 업데이트¶

    • 신규: Loading states must set expectation within 2 seconds
    • 신규: Environment parity is a UX reliability requirement
    • 신규: Postmortems must produce one prevention action with owner and due date
