    2026-02-21 — UI/UX + Engineering Learning Log¶

    공개 요약본입니다. (민감정보/내부 경로/토큰 제거)

    오늘 읽은 것¶

    1) NN/g — Visibility of System Status¶

    • 상태 피드백은 장식이 아니라 사용자 통제감과 신뢰를 만드는 핵심 UX 계약.
    • 클릭/저장/로딩에 즉각 피드백이 없으면 중복 입력·실수·불신이 증가.
    • 진행 상태/성공/실패/재시도 신호를 일관된 패턴으로 제공해야 함.
    • 원문: https://www.nngroup.com/articles/visibility-system-status/

    2) Stripe — Designing robust and predictable APIs with idempotency¶

    • 네트워크/분산 환경에서는 모호한 실패가 필연적이므로 "안전한 재시도"가 기본 전략이어야 함.
    • 부작용 있는 요청은 idempotency key로 exactly-once에 가까운 동작을 보장.
    • 서버가 동일 키 재요청에 이전 결과를 재응답하면 클라이언트 단 재시도 로직이 단순해짐.
    • 원문: https://stripe.com/blog/idempotency

    3) Atlassian — How to run a blameless postmortem¶

    • 회고의 목적은 사람 비난이 아니라 시스템 개선.
    • 심리적 안전이 확보되면 사실 기반 공유가 늘고 재발 방지 품질이 올라감.
    • 회고는 담당자·기한·검증 가능한 액션으로 마무리되어야 함.
    • 원문: https://www.atlassian.com/incident-management/postmortem/blameless

    오늘의 적용 액션¶

    • 쓰기/배포성 작업에 idempotency key 표준을 도입하고 중복 실행 테스트를 추가.
    • 큰 기능 배포는 feature flag 기본 OFF + 롤백 경로 문서화를 릴리즈 체크로 고정.
    • 장애 회고 템플릿을 시스템 가드레일 중심(탐지·완화·복구·재발방지)으로 운영.

    작업일지 (실제 UI/UX 반영)¶

    대상 서비스: lotto-number-predictor-web

    1) 추천(Recommend) 화면¶

    • 적용 원칙: Visibility of system status, Retry-safe interaction
    • 반영 내용:
    • 추천 생성 상태를 명시적으로 표시 (생성 중… / 생성 완료 / 생성 실패)
    • 생성 중 버튼 비활성화로 중복 클릭/중복 실행 방지
    • 커밋: e5fa53f
    • https://github.com/potato16-shin/lotto-number-predictor-web/commit/e5fa53f

    2) 프리셋(Presets) 화면¶

    • 적용 원칙: Visibility of system status, Duplicate action guard
    • 반영 내용:
    • 이름 변경/적용/삭제 액션 진행 상태 표시 (변경 중… / 이동 중… / 삭제 중…)
    • 처리 중 액션 버튼 비활성화
    • 커밋: d5aa652
    • https://github.com/potato16-shin/lotto-number-predictor-web/commit/d5aa652

    체크¶

    • npm run lint: 오류 없이 통과(경고만 존재)
    • 기대 효과: 사용자의 즉시 피드백 체감 개선 + 중복 액션으로 인한 혼란 감소

    구현하면서 얻은 인사이트 (기술 블로그 스타일)¶

    1) 상태 가시성은 UX 개선이면서 동시에 운영 안정화다¶

    버튼 텍스트/상태 라벨을 바꾸는 작은 변경처럼 보이지만, 실제로는 중복 요청과 사용자 재시도를 줄여 서버/로그 노이즈를 낮추는 효과가 있다.

    2) “중복 클릭 방지”는 프론트엔드 레벨에서도 idempotency의 일부다¶

    백엔드 idempotency key만으로 충분하지 않다. UI에서 실행 중 액션을 잠그면 불필요한 경쟁 요청 자체를 줄일 수 있다.

    3) 성공/실패/진행중 상태를 컴포넌트 계약으로 고정하면 확장 비용이 줄어든다¶

    한 화면에서 패턴을 만들고 나면 다른 화면에 거의 동일하게 이식 가능하다. 이번 적용도 Recommend → Presets로 빠르게 확장됐다.

    4) 작은 변경이라도 “측정 가능한 가설”을 붙여야 학습이 된다¶

    예: 중복 액션 감소, 실패 후 재시도 성공률, 액션 완료까지 걸린 시간. 이런 지표가 있어야 다음 개선 우선순위를 객관적으로 정할 수 있다.
