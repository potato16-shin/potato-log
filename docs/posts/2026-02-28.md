## 오늘 읽은 것

### 1) NN/g — Progressive Disclosure
- 핵심 기능은 첫 화면에, 드문 고급 기능은 2차 레이어로 분리할수록 학습 부담과 오류율이 줄어든다.
- 핵심/고급 분리는 ‘중요도’만이 아니라 실제 사용 빈도로 검증해야 한다.
- 웹/모바일 모두에서 초기 인지 부하를 낮추는 기본 패턴으로 유효하다.
- 원문: <https://www.nngroup.com/articles/progressive-disclosure/>

### 2) Martin Fowler — Ship / Show / Ask
- 모든 변경을 같은 PR 정책으로 다루기보다 변경 성격에 맞게 Ship/Show/Ask를 선택하면 리드타임과 피드백 품질이 함께 개선된다.
- Show는 빠른 통합 + 사후 피드백 대화를 가능하게 한다.
- Ask는 불확실성이 큰 변경에 검토 대기 시간을 의도적으로 투자하는 방식이다.
- 원문: <https://martinfowler.com/articles/ship-show-ask.html>

### 3) ProductPlan — RICE Scoring Model
- Reach/Impact/Confidence/Effort를 함께 다뤄야 우선순위 판단의 편향을 줄일 수 있다.
- Confidence를 점수에 넣지 않으면 근거 약한 아이디어가 과대평가되기 쉽다.
- Confidence가 낮은 항목은 즉시 개발보다 빠른 검증 루프로 분리하는 편이 안전하다.
- 원문: <https://www.productplan.com/glossary/rice-scoring-model/>

## 오늘의 적용 액션
- 옵션이 많은 화면은 기본 경로/고급 경로를 분리해 첫 성공까지의 단계를 축소
- 코드 변경 단위별로 Ship/Show/Ask 운영 기준을 도입해 리뷰 병목 완화
- UI 개선 백로그에 RICE + Confidence 기준을 붙여 실행 순서 명확화
